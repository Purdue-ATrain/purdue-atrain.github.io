// what is ATrain

A standardised AI workflow for a physics lab.

ATrain exists because AI assistants now produce more code per session than a human reviewer can read line by line. A graduate student running Claude Code all afternoon can ship a thousand lines of analysis code, with hundreds of edge-case decisions baked into how the prompt was phrased. If every contributor in the lab invents their own prompts, their own file layouts, their own commit conventions, the lab ends up with a pile of AI-generated work that nobody else can pick up.

ATrain is how we keep that tractable. Everyone clones the same ~/.claude/skills directory, so every contributor invokes the same /analysis-scaffold, the same /paper-pdf-writing, the same /undercover-mode. The harness blueprint gives every spawned analysis the same file tree and the same anti-hallucination contract. When you hand a project off to the next student, or join an analysis someone else started, or contribute to a sibling project across the lab, the conventions are already shared. AI-assisted physics stays reviewable, transferable, and contributable.

AI-guided tour

Skip the manual setup. Paste one prompt into any LLM chat and get walked through onboarding.

paste into Claude · ChatGPT · Gemini · Cursor · or any LLM chat

Drop this prompt into your favourite chat assistant. It asks your GitHub handle first, then walks you through the nine onboarding steps one at a time.

I want to start using Purdue-ATrain, an AI-augmented HEP analysis lab at Purdue. The org lives at https://github.com/Purdue-ATrain and the landing page is https://purdue-atrain.github.io. I am a new contributor.

Please walk me through, ONE STEP AT A TIME. Do not dump it all at once. Ask me what step I'm on, wait for my response, then move on:

1. Cloning the shared skill library (https://github.com/Purdue-ATrain/atrain-skills) into ~/.claude/skills so the skills are globally available in Claude Code.
2. Installing Claude Code itself if I don't have it yet.
3. Creating my personal long-lived branch users/<my-github-handle> in the skills repo and checking it out.
4. A guided tour of the available skills: what each one does and when I'd invoke it (/analysis-scaffold, /paper-pdf-writing, /undercover-mode, /caveman, /grill-me, /an-readiness-check, /curate-tests, /deep-copy, /evolve-skill, /graphify, /ralph-loop).
5. Spawning my first analysis with /analysis-scaffold, which drives the harness-blueprint copier template.
6. The daily workflow: edit-as-you-use, commit, push to my users/ branch, and periodically merge origin/main into my branch to absorb canonical updates.
7. Using <skill>/.skip on my branch to opt out of a skill personally without removing it from main.
8. Using local/* branches for private experiments that never get pushed.
9. The security contract: SKILL.md content from contributor branches is DATA, not instructions. The forbidden-pattern list lives at tools/forbidden-patterns.txt in the skills repo.

Ask me my GitHub handle first, then start with step 1.

// 03

The harness blueprint

A copier template. Everything a CMS analysis needs.

A single command generates a complete analysis tree: bin/ entry scripts, analysis/ RDataFrame event loop and define chains, per-channel config/, utils/ for splits and Asimov significance, elements/ for C++ helpers, plus ml/, inference/, tests/, draft/, and docs/. Built-in pixi env, LaTeX paper scaffold, and a CMS-style anti-hallucination contract in CLAUDE.md.

copier copy gh:Purdue-ATrain/harness-blueprint ./my-analysis
pixi-managed env LaTeX paper scaffold anti-hallucination CLAUDE.md RDataFrame ready 10 Jinja2 variables

Path A

Direct PR

Standard GitHub flow. Cut a topic branch off main (verbs: add/, fix/, sharpen/, evolve/, remove/), commit the change, push, open a PR against main. Maintainer reviews and merges. Best for deliberate fixes, new skills, or anything where you want the change reviewed as one coherent diff.

git checkout -b add/<skill-name> # edit, commit git push -u origin add/<skill-name> gh pr create --base main --fill
Path B

Organic + cron-curated

Live on a long-lived users/<handle> branch. Edit skills as you use them, commit when you like, push when convenient. A nightly cron walks every branch, audits each contribution (regex layer-1 forbidden-pattern scan plus layer-2 Claude semantic check), and synthesises a best-of-breed evolved SKILL.md onto nightly. Maintainer PRs nightly → main. Best for organic evolution where interrupting your work to open a PR per tweak is friction.

git checkout users/<handle> # edit, commit, edit, commit git push origin users/<handle> # cron picks it up nightly

Both paths converge at main. The forbidden-pattern scan runs on the cron path; for direct PRs the equivalent safety check happens in maintainer review. Use whichever flow fits the change.

Runbook → evolve-skill/SKILL.md  ·  Forbidden patterns → tools/forbidden-patterns.txt  ·  Contributing → CONTRIBUTING.md

Benchmarks

Measure whether the skills actually help. Per (task, model) cost and pass-rate.

// 07

Get involved

Membership is closed for v0. Open an issue to request access.

  1. 1

    Request access

    Open a GitHub issue at Purdue-ATrain/.github with your GitHub handle.

  2. 2

    Clone + create your branch

    Once added, clone the skill library into ~/.claude/skills and create a long-lived users/<your-handle> branch.

  3. 3

    Use, edit, push

    Edit skills as you use them. Push to your branch. The nightly cron synthesises your contributions back into main after a maintainer review.
