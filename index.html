// what is ATrain

A standardised AI workflow for High Energy Physics.

ATrain exists because AI assistants have accelerated Physics developement tremendously. A graduate student running Claude Code all afternoon can ship a thousand lines of analysis code now, with hundreds of decisions baked into how the prompt was phrased. If every HEP contributor invents their own prompts, their own file layouts, their own commit conventions, the group ends up with a pile of AI-generated work that nobody else can pick up. The problem repeats across handovers, across analyses, across collaborations.

ATrain is one answer. Everyone clones the same ~/.claude/skills directory, so every contributor invokes the same /analysis-scaffold and the same /paper-pdf-writing. The harness blueprint gives every spawned analysis the same file tree and the same anti-hallucination contract. When you hand a project off to the next student, join an analysis someone else started, or contribute to a sibling effort across HEP, the conventions are already shared. AI-assisted physics stays reviewable, transferable, and contributable.

Purdue-ATrain is the first instantiation. The pattern generalises to any HEP group running AI-augmented analyses.

Sharpen your skills

Two contribution paths. Same destination: main.

Skills evolve through two paths: a deliberate direct PR off main, or an organic users/<handle> branch that a nightly cron audits and synthesises onto nightly. Both converge at main.
See how the two paths work ↗

// good practice

Sharpen tools for everyone.

Skills are shared. When you improve a skill, you're improving it for the whole group. Don't keep your sharpening private. Push your users/<handle> branch so the cron picks it up, or open a direct PR for a deliberate change.

But don't over-sharpen. If your edit fundamentally changes what a skill does (what its description claims it triggers on, what category of output it produces), that's a new skill, not a modification. Sharpening a spoon into a knife leaves you with neither a working spoon nor a clean knife. Add a new skill directory at that point.

// 04

Multi-agent, when it earns it

Most "agents" should be workflows. The few that should fan out, fan out for a reason.

The useful split is not "AI vs agents", it is workflow vs agent: in a workflow your code decides the steps; in an agent the model decides them. Most things worth building are workflows, and that is a feature. An agent earns its autonomy only when the steps genuinely cannot be enumerated ahead of time.

● all agents converge on one goal
dispatched · working · returning

the orchestrator emits agents; how they move is the pattern: swarm to one, fleet to many, fleet-swarm in groups.

The shapes

topologywhat it iswhen it fits
single agentone loop, one context, all toolsthe default; anything write-heavy
pipelinefixed sequential stages, output to inputcleanly decomposable, predictable steps
routerclassify, dispatch to one handlerenumerable categories
orchestrator-workera lead decomposes at runtime, delegatesopen-ended; can't predict the subtasks
parallel fan-outsame op across many items, then aggregateembarrassingly parallel, latency-bound

Three ways an agent starts

on-demand

You invoke it

A crew you spawn now for the task in front of you.

scheduled

It runs on a clock

The nightly skill-curation cron; a grid-status check every few minutes.

event-driven

A signal fires it

A job finishes, a PR opens, CI goes red. It reacts, reports, escalates.

Scheduled and event-driven agents stay on the loop: they report and escalate, a human approves anything that writes. Autonomy on the trigger, not on the merge.

// say what you mean
Swarm
Many agents converging on one goal.
Fleet
Many agents, each on a different goal, across separate repos in parallel: one per analysis, per stream, per docs.
Fleet-swarm
Groups of agents, each group on its own goal, across repos in parallel. A fleet of swarms.

// 08

The harness blueprint

A copier template. Everything a CMS analysis needs.

A single command generates a complete analysis tree: bin/ entry scripts, analysis/ RDataFrame event loop and define chains, per-channel config/, utils/ for splits and Asimov significance, elements/ for C++ helpers, plus ml/, inference/, tests/, draft/, and docs/. Built-in pixi env, LaTeX paper scaffold, and a CMS-style anti-hallucination contract in CLAUDE.md.

copier copy gh:Purdue-ATrain/harness-blueprint ./my-analysis
// inputs · copier.yml what you describe 
analysis_name: hh-vbf-bbww
analysis_title: VBF HH→bbWW (sl)
channels: emu, mumu, ee
primary_trigger: DST_HLT_PFJet120
physics_process: HH_bbww
lead_author_name: J. Doe
lead_author_github: jdoe
motivation: |
  search for VBF-produced di-Higgs
  decaying to bb + WW (semileptonic)
// outputs · template/ what you get 
hh-vbf-bbww/
├─ README.md
├─ CLAUDE.md            # anti-halluc.
├─ pixi.toml + pixi.lock
├─ bin/                 # entry scripts
├─ analysis/            # runner, defs
├─ config/              # channels.py
├─ utils/, elements/
├─ ml/, inference/, tests/
└─ draft/, docs/, tools/
pixi-managed env LaTeX paper scaffold anti-hallucination CLAUDE.md RDataFrame ready 10 Jinja2 variables

Get involved

Membership is closed for v0. Open an issue to request access.

  1. 1

    Request access

    Open a GitHub issue at Purdue-ATrain/.github with your GitHub handle.

  2. 2

    Clone + create your branch

    Use, edit, push

    Edit skills as you use them. Push to your branch. The nightly cron synthesises your contributions back into main after a maintainer review.
